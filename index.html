The Transformer — How LLMs Actually Work
~6-8 hours
Goal: Get the architectural foundation. Understand attention, self-attention, and the encoder-decoder structure that every modern LLM (and therefore every agent) is built on.
Attention Is All You Need (Vaswani et al., 2017)
Paper
arxiv.org/pdf/1706.03762

The original Transformer paper from Google Brain that replaced recurrent and convolutional sequence models with pure attention. Introduces the encoder-decoder architecture with stacked self-attention layers, scaled dot-product attention, multi-head attention (8 parallel heads in the base model), and sinusoidal positional encodings. The base model achieved 28.4 BLEU on English-to-German translation while training in 12 hours on 8 GPUs — a fraction of prior state-of-the-art training cost.

Core concepts to internalize

  • Self-attention: how every token attends to every other token in one step
  • Scaled dot-product attention: Attention(Q,K,V) = softmax(QKᵀ/√dk)V
  • Multi-head attention: parallel attention layers attending to different representation subspaces
  • Positional encoding: how the model knows token order without recurrence
  • Why this architecture parallelizes better than RNNs/LSTMs
⏱ 3-4 hours (dense, re-read sections 3-4)
Dissociating Language and Thought in Large Language Models (Mahowald et al., 2023)
Paper
arxiv.org/pdf/2301.06627

A cognitive-science perspective on what LLMs are actually good at — and what they're not. The authors (from UT Austin, Georgia Tech, UCLA, MIT) distinguish formal linguistic competence (knowing grammar and patterns) from functional linguistic competence (using language to reason, plan, and act in the world). They ground this in human neuroscience: the two rely on different brain networks. LLMs are surprisingly strong on formal competence but spotty on functional competence, which often requires fine-tuning or external modules.

Why this matters for agents

  • Agents need functional competence — reasoning, planning, world-modeling — not just fluency
  • This paper explains why raw LLMs alone struggle with agentic tasks
  • It justifies the need for tool use, external memory, planning loops, and RAG
  • Sets up the rest of the course: everything we'll build is essentially scaffolding around the functional-competence gap

Priority techniques for agent-builders

  • Chain-of-Thought (CoT) — get the model to reason step by step before answering
  • ReAct — Reason + Act loop: the foundational pattern for almost every agent
  • Reflexion — adding a self-critique step after each action
  • RAG — grounding the model in external documents before it answers
  • Prompt Chaining — breaking a complex task into a pipeline of smaller prompts
  • Function Calling — structured tool use via JSON schemas

Planning & Reasoning for Agents
~3-4 hours
Goal: Understand the hardest problem in agentic systems — planning. Learn why off-the-shelf LLMs struggle with multi-step tasks, and the architectural patterns (general-purpose vs domain-specific cognitive architectures) that production agents actually use.
Planning for Agents (Harrison Chase, LangChain)
Blog
langchain.com/blog/planning-for-agents

The fourth installment in LangChain CEO Harrison Chase's "In the Loop" series. Distinguishes short-term action selection (handled well by function calling) from long-horizon planning (still hard). Covers the lowest-hanging fixes (more context, better prompts) and then the bigger idea: cognitive architectures — the data-engineering logic around the LLM. Contrasts general-purpose patterns (Plan-and-Solve, Reflexion) with domain-specific cognitive architectures (like the AlphaCodium "flow engineering" approach for code generation). Concludes that nearly every production agent uses a custom domain-specific architecture — and that's why LangGraph exists.

Key takeaways

  • Function calling = short-term action choice; planning = the longer-horizon sequence
  • Long context windows can hurt performance as the agent gets "distracted"
  • General-purpose architectures (Plan-and-Solve, Reflexion) help, but rarely enough for production
  • Domain-specific cognitive architectures move planning work from the LLM into the engineer's code
  • Code is a great way to communicate "what the agent should do" — sometimes better than a prompt

What is LangGraph? (IBM Think)
Article
ibm.com/think/topics/langgraph

An accessible overview of LangGraph, the open-source agent framework built by LangChain. Uses a "Super-Map / Navigator / Cartographer" metaphor to explain graph-based agent architectures. Breaks down the key components: stateful graphs (each node is a step that remembers prior state), cyclical graphs (loops, essential for agent runtimes), nodes (agents or tool calls), edges (conditional branches between nodes), human-in-the-loop, RAG integration, and LangSmith for managing LLMs inside the graph. Covers scaling via multi-agent workflows, real use cases (chatbots, agent systems, LLM applications, Norwegian Cruise Line's guest-facing AI), and LangGraph Studio's visual interface for non-coders.

Concepts to master

  • State — the memory bank that records everything the agent has done
  • Cyclical graphs — why loops matter (a ReAct agent is literally a loop)
  • Nodes & Edges — how to compose tools, models, and conditional logic into a workflow
  • Human-in-the-loop — pausing the agent for human review at critical steps
  • Multi-agent collaboration — routing tasks across specialized agents in parallel
  • Where LangGraph fits next to LangChain, CrewAI, AutoGen, MetaGPT

Capstone: Build Your First Agentic System

Apply everything from weeks 1-6 to a real project. Pick a domain you know well — that's where domain-specific cognitive architectures (Week 5) will pay off most.

  • Pick an LLM justified by Week 3's criteria (reasoning, context, cost, agent-readiness)
  • Write prompts using Week 1 best practices and Week 4 advanced techniques (CoT, ReAct, RAG)
  • Design a cognitive architecture (Week 5) — not just one giant prompt
  • Implement it as a LangGraph with stateful nodes, conditional edges, and at least one tool call
  • Add a human-in-the-loop checkpoint at a high-stakes step
  • Evaluate it: build a small eval set and measure success rate before and after each change
